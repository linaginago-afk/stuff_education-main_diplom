# АИС ПУОП «Теремок» (MVP)

## Запуск локально
```bash
python -m venv --clear .venv

.venv\Scripts\activate         

python -m pip install -r requirements.txt

# Windows PowerShell: корректный вывод путей с кириллицей
$env:PYTHONIOENCODING = "utf-8"

# Быстрый локальный запуск без MySQL
$env:DATABASE_URL = "sqlite:///./teremok.db"

# Для MySQL вместо SQLite укажите свой DATABASE_URL:
# $env:DATABASE_URL = "mysql+pymysql://user:pass@localhost/dbname"

python -m uvicorn backend.main:app --reload
```
Откройте `http://localhost:8000` — попадёте на форму логина.

Стартовый администратор создаётся автоматически:
- логин `admin`
- пароль `admin`

## Импорт сотрудников
Администратор может создать учетные записи сотрудников массово на странице `Сотрудники`.

Поддерживаются файлы `.csv` и `.xlsx` со строкой заголовков:
- `ФИО`
- `email`
- `пароль`
- `подразделение`

Email используется как логин сотрудника. Все сотрудники из файла создаются с ролью `employee`; строки с ошибками или дублирующимися email пропускаются и показываются в отчете импорта.

## Тестовые данные
Чтобы быстро заполнить подключенную базу демо-данными, используйте seed-скрипт:

```bash
python -m backend.seed --employees 50 --reset-seed
```

Скрипт создает 50 сотрудников, несколько тестов, вопросы, назначения и результаты прохождения. Повторный запуск без `--reset-seed` не дублирует уже созданных seed-сотрудников и seed-тесты. Опция `--reset-seed` удаляет только ранее сгенерированные seed-данные, не трогая обычных пользователей и тесты.

Пример тестового сотрудника:
- логин `seed.employee001@example.com`
- пароль `test1234`

## Основные возможности
- **Авторизация** через JWT, разделение ролей `employee` / `admin`.
- **Сотрудник**:
  - просмотр назначенных тестов (статус, последние результаты);
  - прохождение тестов (радиокнопки, один правильный вариант), фиксация попытки и подсчёт %;
  - просмотр личного прогресса: лучшая оценка, число попыток, дата последней.
- **Администратор**:
  - управление пользователями (создать, обновить, удалить, роли/подразделение);
  - управление тестами (создать/удалить, добавить/обновить вопросы и варианты);
  - назначение тестов сотрудникам;
  - просмотр результатов: сводка по тесту/сотруднику, лучшая попытка, количество попыток, зачёт/незачёт, детали попыток.

## Структура
- `backend/` — FastAPI, SQLAlchemy модели и роуты (`/api/...`).
- `frontend/` — статические страницы (login, employee, admin) + лёгкий JS (`fetch` к API).
- `requirements.txt` — зависимости бэкенда.

## API 
- `POST /api/auth/login` — получить токен.
- `GET /api/auth/me` — текущий пользователь.
- Админ: `/api/admin/users`, `/api/admin/tests`, `/api/admin/tests/{id}/questions`, `/api/admin/tests/{id}/assign`, `/api/admin/results`, `/api/admin/users/{id}/attempts`.
- Сотрудник: `/api/employee/tests`, `/api/employee/tests/{id}`, `/api/employee/tests/{id}/submit`, `/api/employee/progress`.

## Типовые сценарии (use cases)
- Сотрудник заходит, видит список назначенных тестов, проходит тест «Охрана труда», получает результат 80% и статус «зачёт».
- Сотрудник открывает «Мой прогресс» и смотрит, по каким тестам лучший балл выше порога, сколько было попыток и когда последний раз проходил.
- Администратор добавляет нового сотрудника, создаёт тест «Пожарная безопасность», добавляет 3 вопроса, назначает тест группе сотрудников.
- Администратор открывает «Результаты», фильтрует по тесту и видит сводку: у кого зачёт/незачёт, лучшая попытка, число попыток; при клике — история попыток конкретного сотрудника.
